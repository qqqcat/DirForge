# DirForge 项目综合评估报告（2026-03-16）

## 1. 评估范围与方法

本轮评估覆盖以下内容：

1. Workspace 架构与 crate 边界（`crates/*`）
2. 核心链路：扫描、聚合、缓存、报告、平台能力、删除执行、UI 集成
3. 文档与代码一致性
4. 构建与测试验证：
   - `cargo check --workspace`
   - `cargo test --workspace`

> 评估目标：给出当前可交付能力、主要风险与近期优先级，并同步所有关键文档。

---

## 2. 总体结论

DirForge 当前处于 **Production Readiness（生产就绪冲刺）** 阶段，已经具备稳定的“扫描 -> 分析 -> 展示 -> 删除 -> 导出 -> 诊断”主链路。

### 2.1 当前已达到的能力

- **扫描链路稳定**：目录扫描已采用 worker 池 + 聚合线程 + 有界发布队列，能够处理大目录、取消扫描与错误上报。
- **完成态稳定**：扫描结束阶段不再因完整树复制或重任务串行收尾而假死。
- **结果展示贴近用户目标**：概览、最大文件夹、最大文件、treemap 与最近发现列表成为主视图核心。
- **删除流程回归主界面**：危险动作不再分散在孤立页面，而是进入右侧 Inspector，支持回收站删除和永久删除。
- **删除后即时局部刷新**：删除成功后，Inspector、排行榜、概览统计与 treemap 都会同步更新，而不是等下一次重扫。
- **删除期间可感知**：删除已改为后台执行，确认窗会立即关闭，并通过顶部横幅、Inspector 任务卡片与状态栏持续提示，降低“大文件删除像卡死”的主观感受。
- **目录上下文分析增强**：选中文件夹后，“最大文件”榜单会切换到该目录内部的最大文件，分析不再始终停留在整盘维度。
- **首次体验改善**：默认根路径已优先落到系统盘/首个卷，Overview 提供盘符快捷扫描按钮。
- **文档化程度较高**：安装、使用、架构、评估与进展记录均已具备基础维护结构。

### 2.2 当前仍需重点补强的方向

- **真实删除用户反馈**：永久删除虽然已改为后台任务模式，但删除中仍缺少更细粒度的阶段或字节级进度。
- **跨平台真实删除测试矩阵**：权限不足、文件被占用、系统目录等边界场景仍需更多实机覆盖。
- **UI 自动化回归**：当前主要依赖人工发现问题，像重复 widget ID、滚动区交互异常这类问题缺少自动化保护。
- **回收站可见性体验**：Windows 上已增加回收站二次校验，但仍缺少自动刷新/定位系统回收站的引导动作。

## 4. 分层能力评估

## 4.1 Scan（`dirforge-scan`）

**现状：**
- 使用 `Progress / Batch / Snapshot / Finished` 事件模型
- worker 并发扫描 + 聚合线程处理乱序父子节点
- 发布链路支持有界队列、节流与中间态丢弃，避免 UI 被事件洪泛拖死

**评价：**
- 已从“原型可跑”进入“可验证、可回归的工程实现”

## 4.2 Core（`dirforge-core`）

**现状：**
- `NodeStore` + `rollup` + Top-N 查询稳定
- 现已承担 UI 局部刷新的核心数据来源

**评价：**
- 是当前系统最稳定的共享域模型层

## 4.3 Actions（`dirforge-actions`）

**现状：**
- 支持删除计划构建、真实删除执行、失败分类与审计记录
- UI 已直接消费真实删除执行结果
- Windows 回收站删除增加二次校验，避免“调用成功但系统回收站不可见”时误报成功

**评价：**
- 已达到“可真实执行”的程度，但删除中的用户感知和实机边界覆盖仍需增强

## 4.4 Cache / Report / Telemetry

**现状：**
- SQLite 承担设置、历史、审计
- 诊断包、摘要 JSON、错误 CSV 等导出链路可用
- 遥测和系统快照可进入诊断输出

**评价：**
- 工程骨架齐全，适合作为后续排错与发布诊断基础

## 4.5 UI（`dirforge-ui`）

**现状：**
- 页面结构已稳定为：Overview / Live Scan / Treemap / History / Errors / Diagnostics / Settings
- `Operations` 页面已移除，危险动作进入 Inspector
- 扫描时工具栏语义已切换为 `Stop Scan`
- 榜单已支持独立滚动，且通过独立 widget ID 避免重复控件红框
- 删除后已支持局部树重建，概览统计与 treemap 同步变化
- 永久删除已增加确认层，删除结果会提供更明确的撤销/失败提示
- 删除确认后会立即关闭弹窗，并切换到后台任务提示，用户仍可继续浏览结果
- 选中文件夹后，“最大文件”榜单会切换到该目录内部的最大文件

**评价：**
- 已经不是“开发者面板”，而是具备实际用户价值的桌面磁盘分析器雏形

## 5. 风险矩阵

| 风险项 | 级别 | 现状 | 建议 |
|---|---|---|---|
| 真实删除跨平台边界一致性 | 中 | 有执行链路、审计与回收站校验，但实机覆盖仍有限 | 扩展权限/锁冲突/系统目录/回收站可见性测试 |
| UI 缺少自动化交互回归 | 中 | 目前主要靠人工回归 | 增加 UI 集成测试或最小交互脚本 |
| 删除中的用户感知偏粗 | 中 | 有后台任务提示但无细粒度进度 | 增强阶段性反馈与完成后定位/提示 |
| 首次体验仍可更友好 | 低 | 已增加默认根路径和盘符按钮，但视觉层级还可继续优化 | 优化盘符按钮与自定义路径的引导关系 |
| 文档与代码再次漂移 | 低 | 本轮已同步，但后续变更快 | 合并功能时绑定文档检查 |

## 6. 近期优先级建议（未来 2~4 周）

1. **删除链路回归补齐**
   - 增加更接近真实点击流的 UI 自动化验证
   - 补实机测试：权限不足、锁定文件、回收站失败与 Explorer 可见性

2. **首次体验继续优化**
   - 优化盘符按钮与手动路径输入的视觉层级
   - 增加更贴近普通用户的空状态说明与上下文分析提示

3. **长跑稳定性持续验证**
   - 保持大目录扫描长时间运行观察
   - 继续记录资源曲线、删除长耗时行为与 finish line 稳定性

---

## 7. 结论

DirForge 当前已经从“概念原型”进入“可持续迭代的工程化桌面工具原型”阶段。

最关键的正向变化不是单点修 bug，而是主链路已经形成闭环：

- 扫描不会轻易假死
- 完成态不会轻易卡死
- 删除动作不再跳页
- 删除后的 UI 不再等下次重扫
- 文档与代码现已重新对齐

下一阶段的重点不再是“能不能跑起来”，而是把真实删除交互保护、跨平台边界和 UI 自动化回归继续做扎实。
