# DirForge 项目综合评估报告

> 评估对象：当前仓库 `DirForge`（Rust workspace 版本）  
> 评估目的：给出“当前可交付水平、主要风险、短中期改进计划”，用于研发排期与质量管理。

## 1. 结论摘要（Executive Summary）

当前项目已完成从“概念文档”到“可运行原型”的关键跨越，具备：

- 多 crate 工程骨架
- 扫描/聚合/缓存/基础 UI 的端到端链路
- SQLite 结构化缓存与历史记录
- 基础测试闭环（unit + integration）

但整体仍处于 **Pre-Production（预生产原型）**，距离生产级还有明显差距，主要在：

- 扫描与去重算法仍偏简化
- UI 信息架构与性能优化尚未达到百万节点目标
- 平台能力、可观测性、误删安全策略不完整

**建议定位：** 当前可作为“架构验证版/内部 alpha”，不建议直接对外正式发布。

## 2. 评估范围与方法

评估范围：

- 工程结构与模块边界
- 核心功能链路（扫描/缓存/去重/UI）
- 可测试性与质量保障
- 文档完整度与可维护性

方法：

- 基于仓库当前代码与文档的静态审查
- 对照现有 SDD 与 UI 规格文档进行一致性检查
- 从“生产可用性”视角进行风险分层

## 3. 当前实现覆盖度评估

### 3.1 架构与工程结构

**已完成**

- Cargo workspace 与模块化 crate 划分
- `core/scan/cache/ui/dup/report/actions/platform` 基本职责分离

**不足**

- 部分模块仍为最小实现，接口稳定性与扩展点不足
- 领域模型与应用层策略边界有待进一步收敛

**评估等级：B-**

### 3.2 扫描引擎（Scan）

**已完成**

- 后台线程扫描
- 进度事件/快照/完成事件
- 错误收集

**不足**

- 缺少 profile 化并发策略（SSD/HDD/网络盘）
- 事件节流、批处理与背压机制不足
- 增量重扫与目录级失效策略未完整实现

**评估等级：C+**

### 3.3 缓存系统（Cache）

**已完成**

- SQLite schema（快照/历史/错误/设置）
- 快照读写与历史记录

**不足**

- 缺少 schema version + migration 机制
- 缺少 WAL/恢复策略说明与压力验证
- FileIdentity 模型（volume_id/file_id）未落地

**评估等级：B-**

### 3.4 重复文件检测（Dup）

**已完成**

- 按 size 分组的候选检测

**不足（关键）**

- 未实现 partial hash / strong hash 分阶段收缩
- 结果风险分级与 keeper 推荐仍缺失
- I/O 限流、优先级调度未实现

**评估等级：C**

### 3.5 UI 与交互

**已完成**

- 原生 `egui/eframe` 框架接入
- 页面骨架：Dashboard/Current Scan/History/Error/Settings
- 多语言基础切换（系统语言默认 + 手动覆盖）

**不足（关键）**

- 仍偏“管理面板式”最小界面，未达到完整 IA 目标
- 缺少 treemap 主视图、虚拟化列表、局部失效刷新
- 缺少操作中心/风险确认流程等生产级 UX

**评估等级：C+**

### 3.6 测试与质量保障

**已完成**

- 核心单元测试（rollup、dup）
- 扫描集成测试（fixture）

**不足**

- 性能基准、回归测试矩阵不足
- UI 交互测试缺失
- Windows 平台专项测试不足

**评估等级：B-**

## 4. 风险矩阵（Top Risks）

| 风险 | 影响 | 概率 | 当前等级 | 建议 |
|---|---:|---:|---:|---|
| 扫描性能在大数据量下退化 | 高 | 高 | 高 | 引入分阶段 pipeline + 事件节流 + 批处理 |
| 去重误报/漏报 | 高 | 中 | 高 | 实施 size→partial hash→full hash |
| UI 在大列表下卡顿 | 高 | 高 | 高 | 列表虚拟化 + 增量渲染 |
| 缓存 schema 演进成本高 | 中 | 中 | 中 | 增加 schema_version 与迁移框架 |
| Windows 平台行为不一致 | 中 | 中 | 中 | 增加 windows-rs 能力封装与专项测试 |

## 5. 与目标架构一致性评估

与 `docs/dirforge-sdd.md` 对照后，当前状态为：

- **已对齐**：模块拆分方向、事件驱动基本形态、SQLite 持久化方向、多页 UI 骨架
- **部分对齐**：扫描流水线阶段化、历史记录与错误中心
- **未对齐**：高性能并发模型（IO/CPU 分池细化）、去重四阶段、treemap 增量渲染、操作中心与误删防护链路

## 6. 里程碑建议（接下来 6~8 周）

### M1（第 1~2 周）：性能与数据管线

- 扫描事件批量化（按批次发送）
- Snapshot coalescing（50~100ms）
- 扫描 profile（SSD/HDD/Network）

### M2（第 3~4 周）：去重系统升级

- partial hash / full hash pipeline
- 候选收缩与结果确认
- keeper 推荐与风险标签

### M3（第 5~6 周）：UI 生产化

- treemap 主视图 + 交互
- 大列表虚拟化（files/duplicates/errors）
- 局部失效刷新与帧预算监控

### M4（第 7~8 周）：发布质量收口

- schema migration
- Windows 专项测试与修复
- 安全删除流程与操作中心
- 诊断页与导出诊断包

## 7. 推荐 KPI（用于跟踪改进效果）

- 冷扫描吞吐：files/s, bytes/s
- 热扫描加速比：warm/cold
- UI 交互延迟（P95）
- 去重误报率（抽检）
- 扫描失败路径占比
- 崩溃率与恢复成功率

## 8. 发布建议

发布分级建议：

- **当前：内部 Alpha（推荐）**
- **下一阶段：外部 Beta（需完成 M1~M3）**
- **正式 GA：需完成 M4 + 稳定性门槛验证**

5. 完善风险删除流程与操作中心。
