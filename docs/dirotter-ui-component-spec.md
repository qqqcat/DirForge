# DirOtter UI 组件规格（2026-03-20）

## 1. 当前实现映射

- 主路径页面：Overview / Live Scan / Result View / Settings
- 二级工具页面：History / Errors / Diagnostics（仅在 `高级工具 / Advanced Tools` 开启后显示）
- 已移除页面：Operations
- 已落地布局：顶部工具栏、左导航、中央工作区、右侧 Inspector、底部状态栏
- 已落地能力：四语言切换（中文/英文/法语/西班牙语）、中文字体回退、人类可读大小/计数格式、轻量结果视图、页面级滚动、对称 gutter
- 已落地动作：Inspector 内 `Move to Recycle Bin` / `Delete Permanently` / `Open File Location`
- 已落地刷新：删除成功后局部刷新榜单、概览统计、treemap 与 Inspector
- 已落地反馈：删除确认窗口会先关闭，再转为顶部横幅、Inspector 任务卡片与状态栏提示
- 已落地引导：默认根路径优先系统盘/首个卷，并提供盘符快捷扫描按钮
- 已落地扫描体验：扫描入口已切换为三档用户模式，不再暴露 `batch / snapshot` 数值控件
- 已落地上下文：选中文件夹后，“最大文件”榜单会切换到该目录范围
- 已落地清理建议：Overview 已出现规则驱动的建议卡、详情窗与安全缓存一键清理流
- 已落地维护动作：Inspector 已提供 `释放 DirOtter 内存 / 清理残留 staging`，Diagnostics 已提供手动快照/摘要/错误导出

产品方向：专业、克制、高信息密度，但必须优先服务“找出最大占用并直接处理”的主目标。

## 2. 布局与尺寸

- 顶部工具栏：56px
- 左导航：188px
- 右 Inspector：300px
- 底部状态栏：26px
- 常规行高：28~32px

### 页面内容区

- 主内容区使用统一最大宽度约束
- 主内容区左右两侧使用显式对称 gutter
- Overview 使用独立于其他页面的更窄最大宽度，避免在 Inspector 存在时横向过满；当前应优先保证视觉对称，而不是把内容撑满
- 页面宽度容器应使用真正的居中内容列，不得依赖左右 spacer 假装居中
- `Overview / Live Scan / Result View / Settings` 使用页面级纵向滚动
- `History / Errors / Diagnostics` 属于高级工具页，默认不应打断主清理路径
- 扫描结束后允许进入一个显式 `整理结果中 / Finalizing` 阶段；该阶段必须保持 UI 可交互，不能把最终快照落库和建议汇总压在 UI 线程上
- 不允许通过固定高度主卡或固定高度榜单来“强行对齐”页面
- 滚动页必须保留底部安全区，避免最后一张卡片紧贴视口底边而产生“被裁掉”的观感
- 卡片容器与提示条在紧凑子布局内绘制前，必须放宽当前 `Ui` 的 clip rect，避免右边框和下边框被裁切

## 3. 页面职责

### Overview

- 负责清理建议、扫描目标、最大文件夹、最大文件、盘符快捷扫描与首页级空间结论
- 首页必须遵循主流 dashboard 的信息顺序：
  - Hero 结论区
  - KPI 指标条
  - 全宽扫描卡
  - 双列证据区（最大文件夹 / 最大文件）
- 首页不得复用 Settings 的 `设置行` 组件语义；Overview 必须使用专用 dashboard 卡片与显式列宽
- 双列区域必须使用显式宽度分配和固定 gutter，避免卡片漂移、重叠和左右留白失衡
- Hero 区必须先回答“现在最值得做什么”，不能把设置页式说明文案堆到顶部
- KPI 指标条应优先展示唯一信息，不得再与独立卷摘要卡重复；当前固定展示 `磁盘已用 / 磁盘可用 / 已扫描体积 / 错误`
- KPI 卡片必须吃满各自分配列宽，不允许因内容本身过窄而缩成左对齐小块
- 扫描目标卡必须优先展示用户模式，而不是技术参数；卷级补充信息应并入卡内紧凑状态条，而不是再复制一张大摘要卡
- 扫描目标卡必须先展示盘符快捷按钮，再展示手动目录输入；手动输入框只作为次级入口，不应以全宽大控件占据卡片顶部
- 扫描完成后，Overview 顶部必须优先展示 `清理建议` 结论与动作入口
- `清理建议详情` 的顶部操作条必须包含可见的删除动作，不允许把主清理按钮埋到长列表底部
- 红色高风险项不得只提示“禁删/Blocked”；必须明确引导用户打开所在位置并手动确认处理
- Hero 必须包含：
  - 可释放空间总量或“暂无明确建议”
  - `查看建议详情`
  - `一键清理缓存（推荐）`
  - 右侧优先来源列表
- `一键清理缓存（推荐）` 不再复用普通回收站删除：
  - 点击后应尽快返回成功反馈
  - 目标先进入应用自己的 staging 区
  - 真正释放空间在后台继续完成
- `查看建议详情` 对话框必须：
  - 居中显示
  - 提供顶部或标题栏关闭入口
  - 底部保留 `关闭`
  - 为路径列和大小列做显式宽度分配，避免长路径把右侧大小截断
  - 提供快捷操作，至少包括：`全选安全项`、`清空所选`、`打开所选位置`
  - 当分类为缓存时，主操作应明确显示为 `快速清理选中缓存`
- 必须显式区分：
  - `Scanned Size`
  - `Volume Used`
  - `Total Capacity`

### Live Scan

- 展示扫描中已发现的最大项与最近扫描到的文件
- 必须清楚表达“这是实时增量视图，不是最终结论”
- 选中文件夹后，“最大文件”榜单应优先展示该目录内部的最大文件，而非始终显示整盘结果
- 排行榜不应只露出一小段可操作区；页面滚动应优先于卡片内部固定高度裁切

### Result View

- 只在扫描完成后展示结果，不跟实时扫描绑定
- 只展示当前目录的直接子项，不递归整树
- 支持逐层进入下一层和返回上级
- 必须优先保证低开销与稳定性，而不是追求复杂布局算法
- 页面布局必须采用“顶部摘要 + 底部填充型结果区”
- 结果区应吃满当前页面剩余高度，长列表由结果区内部滚动承载，不允许出现大面积闲置留白

### History

- 展示历史扫描记录与快照摘要
- 默认隐藏，仅在高级工具开启后显示

### Errors

- 展示错误分类与路径
- 从错误列表选中对象后，应直接在 Inspector 查看和处理
- 默认隐藏，仅在高级工具开启后显示

### Diagnostics

- 展示诊断 JSON 与导出入口
- 额外承担手动维护动作：
  - `Save Current Snapshot`
  - `Record Scan Summary`
  - `Export Errors CSV`
- 默认隐藏，仅在高级工具开启后显示

### Settings

- 展示语言、主题和本地化说明
- 必须提供 `高级工具 / Advanced Tools` 显隐开关，并持久化该选择
- 不得在页面级宽度约束内部再套一层额外固定宽度，否则容易让右侧卡片出现视觉截断
- 设置页改为章节式单列布局：
  - 常用设置（语言与主题）
  - 视觉方向
  - 本地化说明
  - 品牌含义
- 设置页应优先采用“设置分组 + 设置行”的主流结构，而不是说明卡片拼贴
- 高频控制项必须靠上，说明性内容靠下

## 4. 工具栏规范

- 空闲状态：
  - 主按钮：`Start Scan`
  - 次按钮：`Cancel` 可禁用
- 扫描状态：
  - 主按钮必须禁用或隐藏 `Start Scan`
  - 次按钮必须切换为 `Stop Scan`
- 停止扫描被点击后，按钮应进入 `Stopping` 禁用态，直到后台 worker 真正退出并收尾完成
- 不允许在扫描进行中继续显示可点击的 `Start Scan`
- 顶部标题旁的状态胶囊必须使用当前语言实时本地化文本
- 后台删除任务至少要区分：
  - 普通回收站删除
  - 快速清理缓存
  - 永久删除
- 当 Inspector 当前选中项属于安全缓存时，应优先显示 `快速清理缓存`，而不是只给普通回收站删除

## 5. Inspector 规范

- Inspector 始终展示当前全局选中对象
- 选中对象来源可以是：
  - 排行榜
  - Live Scan 最近文件
  - Treemap
  - History
  - Errors
- Inspector 必须包含：
  - 名称/路径/大小
  - 来源上下文
  - 快速操作区

### Quick Actions

- `Open File Location`
- `Move to Recycle Bin`
- `Delete Permanently`
- `Release DirOtter Memory`
- `Clean Up Staging`
- `Cleanup Suggestions` 触发的批量清理也必须复用同一条删除执行链路

要求：

- 危险动作不再进入独立页面
- 永久删除必须先经过确认层
- 点击确认删除后，确认窗口必须立即关闭，再以后台任务形式给出可感知反馈
- 删除成功后，当前选中对象必须从 Inspector 清除
- 删除成功后，相关榜单、概览统计和结果视图必须即时局部刷新
- 回收站删除成功后应提示可撤销
- Windows 回收站删除不应只依赖 API 返回值，需有系统回收站可见性的二次确认
- 删除失败后应给出针对性的失败提示
- 批量清理确认必须明确写清：项目数、预计释放量、默认走回收站

## 6. 排行榜与滚动区规范

- Overview / Live Scan 顶部 Top-N 卡片不得再嵌套固定高度 mini-scroll
- 榜单当前条目应直接进入页面内容流，由外层页面滚动承担浏览
- 列表项点击后应联动 Inspector
- 路径允许中间截断，但必须保留大小与排序信息
- 如果存在独立滚动区，必须拥有独立 widget/scroll ID，避免调试模式下重复 ID 红框

## 7. Result View 规范

1. 结果视图必须是“扫描完成后的结果页”，不能参与实时扫描刷新
2. 默认展示当前目录的直接子项，限制最大展示条目数，避免大目录压垮 UI
3. 点击条目应联动 Inspector；目录项应支持继续进入下一层
4. 如果扫描仍在进行，应明确提示用户等待完成，而不是尝试实时渲染
5. 结果列表必须是填充剩余高度的主浏览区，而不是随内容高度塌缩的小卡片

## 8. 扫描目标与首次引导

- 默认根路径应优先使用系统盘或首个卷挂载点，不得默认停留在 `.`
- 应提供盘符/卷快捷按钮，点击后直接开始扫描对应卷
- 文本输入框保留为高级入口，用于扫描任意目录
- 扫描参数不再直接暴露给普通用户，必须改为模式化选择：
  - `快速扫描（推荐）`
  - `深度扫描`
  - `超大硬盘模式`
- 必须明确说明三档模式都会完整扫描当前范围，只是扫描节奏与界面刷新方式不同
- 空状态提示需明确说明“优先点盘符按钮，也可手动输入目录”

## 9. 本地化与主题

- 默认依据系统语言环境推断语言
- 设置页手动选择优先级高于自动检测
- 当前自动检测覆盖 `zh / fr / es / en`
- 法语与西班牙语必须提供完整 UI 文案覆盖，不得把说明文案或状态提示退回英文
- Windows UI 字体优先采用 `Segoe UI / Segoe UI Variable` 风格，中文继续回退到 `Microsoft YaHei / DengXian / SimHei / SimSun`
- 浅色与深色主题都必须保持完整 panel/surface 对比度，不能出现浅色控件叠黑底

### DirOtter 色彩语义

- 主品牌色：`River Teal`
  - `#2F7F86`
  - Hover：`#276D73`
  - Active：`#1F5C61`
- 深色基础色：`Deep Slate`
  - App background：`#11181C`
  - Panel background：`#182227`
  - Elevated panel：`#1F2C32`
  - Text primary：`#EAF2F4`
  - Text secondary：`#A8BBC1`
  - Accent：`#4BA3AC`
- 浅色基础色：
  - Window background：偏灰白，避免纯白刺眼
  - Surface / divider 基础：`Mist Gray` `#E8EEF0`
  - 主文本：`Deep Slate` `#1E2A30`
- 轻暖辅助色：`Sand Accent` `#D8C6A5`
  - 仅用于空状态、少量品牌点缀或说明性元素，不得大面积覆盖分析界面

## 10. 仍待补强的交互

- 更正式的 12-column 页面栅格
- 更强的视觉回归保护
- 删除中的更细粒度阶段或进度表达
- `Overview / Live Scan / Result View` 更统一的版式节奏

## 11. 清理建议规范

1. 分类先走规则，不依赖 AI。
2. 风险必须直接映射到 UI：
   - `Low` => 绿色 `可清理`
   - `Medium` => 黄色 `谨慎`
   - `High` => 红色 `手动处理 / Manual Review`
3. `一键清理缓存（推荐）` 只允许命中安全缓存项，不得混入 warning / blocked 项。
4. 详情窗中：
   - 绿色默认勾选
   - 黄色默认不勾选
   - 红色锁定不可勾选
5. 详情列表点击路径后必须联动 Inspector，保持“建议 -> 检查 -> 执行”是同一条用户路径。
6. 建议计算不得无限扩大：
   - 每类候选必须有上限
   - 全局总候选必须有上限
   - 优先返回用户当前最可能执行的候选，而不是保留全树原始命中项
