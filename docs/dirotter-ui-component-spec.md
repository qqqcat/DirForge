# DirOtter UI 组件规格（2026-05-02）

## 1. 当前产品结构

- 主路径页面：`Overview / Live Scan / Duplicate Files / Settings`
- 二级工具页面：`Errors / Diagnostics`
- 危险动作入口：右侧 `Inspector`
- 当前产品心智：克制、直接、高信息密度，但一切围绕“帮用户释放空间”

## 2. 页面壳体

固定 shell：

- 顶部工具栏
- 左侧导航
- 中央主内容区
- 右侧 Inspector
- 底部状态栏

布局原则：

- 主内容区必须使用明确的最大宽度与对称 gutter
- Overview 可以使用单独更窄的宽度约束
- 页面优先整体滚动，不依赖固定高度主卡强行裁切内容
- Inspector 必须拥有独立纵向滚动
- 顶部工具栏、左侧导航、右侧 Inspector 和底部状态栏都必须有壳体内边距；任何标题、状态数字或说明文字不得贴住窗口边缘
- 左侧导航只保留品牌与导航项，不放低价值引导句，避免在窄栏里把文案压到边缘
- 宽屏下 Overview 必须使用可用横向空间，不能让主内容区过窄而把左侧或底部信息挤到边缘

## 3. 页面职责

### Overview

- 首页必须先回答“现在最值得做什么”
- 必须优先展示：
  - `一键提速（推荐）`
  - 清理建议摘要
  - KPI
  - 最大文件夹 / 所选范围最大项目证据区
- 扫描入口必须优先展示盘符快捷按钮，手动目录输入是次级入口
- 不能把底层批次、快照、并发参数直接暴露给普通用户
- Top-N 证据区必须以扫描快照为最终数据源，不允许在 UI 帧内再用 `fs::metadata()` 过滤导致结果为空

### Live Scan

- 显示实时进度、实时 Top-N、最近扫描文件
- 必须明确这是“实时增量视图”，不是最终结果
- 左侧显示当前最大文件夹；右侧在有选中目录时显示该目录的直接子项或已知最大项，否则显示全局最大文件
- 停止扫描必须通过 `Stop Scan` 进入安全收尾

### Duplicate Files

- 目标必须是“帮助用户放心删除重复文件”，不是单纯列出重复项
- 检测必须先按 `size` 收敛候选，再在页面进入后后台补算 hash
- UI 必须按组展示，默认按可释放空间降序
- 每组必须自动给出 `推荐保留`
- 默认只删除非推荐项，且高风险组默认不自动选
- 页面必须支持折叠组、懒加载和打开所在位置

### Settings

- 展示语言、主题和 `高级工具` 开关
- 如果当前回退到临时会话存储，必须显式提示本次设置不会跨退出保留

### Errors / Diagnostics

- 默认不打断主路径
- 只在高级工具开启后显示
- Diagnostics 当前只展示当前会话结构化信息和维护动作

## 4. Inspector 规范

Inspector 始终服务当前选中对象。

必须包含：

- 名称
- 路径
- 大小
- 快速操作
- 当前后台任务或最近执行反馈
- 系统内存状态摘要

当前关键动作：

- `Open File Location`
- `Move to Recycle Bin`
- `Delete Permanently`
- `Fast Cleanup`
- `Release System Memory`

动作约束：

- `Fast Cleanup` 只能在低风险缓存目标上出现，不能长期以灰按钮占位
- 永久删除必须先过确认层
- 删除确认后窗口必须立即关闭，反馈转为后台任务态
- 删除失败的完整细节必须收口到详情视图，不得挤在外层 banner
- 普通按钮、主按钮、危险按钮必须显式满足明暗主题下的文字/背景对比；危险动作统一为白字红底，不能依赖全局文本色自动推断
- 禁用按钮仍必须可读：禁用态可以降低强调程度，但不能出现深色底叠深色字或浅色底叠浅色字

## 5. 删除与反馈规范

- 后台删除线程必须逐项回传已处理、成功、失败统计
- 删除完成后的结果同步必须在后台进行
- `结果同步中` 必须是显式可见状态
- 回收站删除成功后应提示用户可从系统回收站恢复
- Windows 回收站删除要做可见性二次确认

## 6. 清理建议规范

- 必须是规则驱动，不依赖 AI 推断
- 风险映射必须直接进入 UI：
  - `Low` => 可清理
  - `Medium` => 谨慎
  - `High` => 手动处理
- 绿色项可默认勾选
- 黄色项默认不勾选
- 红色项必须锁定，不允许直接执行
- 详情窗必须支持完整列表审阅，而不是只显示前几项
- 低风险缓存候选必须覆盖常见 Windows 缓存根，包括 `Temp`、`Cache`、`.cache`、`LocalCache`、`INetCache`、`__pycache__`
- `Fast Cleanup` 删除完成后不得为了刷新 UI 体验强制重建完整结果树；应先更新摘要、Top-N 和清理建议，完整树仅在用户进入需要它的页面时后台载入

## 7. 结果与恢复规范

- 结果默认只保留在当前会话内
- 完成态浏览收口到 Overview 的最大文件夹/文件证据区与 Inspector
- 删除/结果同步期间，若结果树未驻留内存，依赖完整树的页面应显示等待态
- 不允许在渲染函数内同步读取快照、解压快照或反序列化 `NodeStore`

## 8. 本地化与字体

- 当前支持 19 种语言选择
- `中文 / English / Français / Español` 为完整 UI 文案
- 其余语言当前允许英文回退
- 字体 fallback 不能一次性加载整组 Windows 大字体；默认只加载主 UI 字体、CJK 主要 fallback，以及必要的小体积脚本 fallback
- 如后续恢复更多脚本专用字体，必须先量化启动工作集增长，不能把几十 MB 的 TTC/TTF 全部常驻

## 9. 当前仍需补强

1. 最小视觉回归：Overview 明/暗主题、左侧导航、右侧 Inspector 操作按钮、底部状态栏、Settings 下拉框都要纳入截图或手测门禁
2. 更统一的页面节奏与细部栅格
3. 删除过程更强的阶段反馈
4. 多语言下的长期回归保护

## 10. 视觉回归排查顺序

遇到布局或主题回归时，先按截图事实收紧约束，再查最终绘制面：

1. 壳体边距：`toolbar_frame`、`panel_frame`、`statusbar_frame`、`CentralPanel` inner margin
2. 页面宽度：`with_scrollable_page_width`、`with_page_width_fill_height`、Overview 最大宽度
3. 按钮可读性：`sized_button`、`sized_primary_button`、`sized_danger_button` 以及直接构造的危险按钮
4. 主题色：`apply_theme` 必须先重建 `Visuals::light()` / `Visuals::dark()`，再覆盖 palette；不能只改颜色不改 `dark_mode`
5. 低级绘制坐标：`Painter` 路径必须以 `ui.cursor().min` 作为内容起点，不能用 viewport 坐标当绝对绘制坐标
6. 明暗主题各跑一次手测；确认导航标题、底部数字、Settings 下拉框、Inspector 按钮文本都不贴边、不混色

## 11. 内存占用策略

- 启动期：只加载必要字体 fallback，避免一次读入多个十几 MB 的 Windows TTC/TTF。
- 扫描期：实时列表只作为最近扫描文件展示，不作为完整结果存储。
- 扫描完成：保留摘要、清理建议和 Top-N 证据；完整 `NodeStore` 写入会话快照后释放内存。
- 重复文件页需要完整结果树时再后台载入快照，页面必须显示等待态，不能在 UI 当前帧同步解压。独立 Result View / Treemap 入口已删除。
- 当前不引入 SumTree 作为短期修复项。扫描端已有增量聚合、固定容量 Top-K、字符串池和完成后释放 `NodeStore` 的内存策略；SumTree 只有在需要长期驻留、跨会话增量编辑或复杂范围查询时才进入重构候选。
